Auditly — Marketing Site + Dashboard (Next.js) · Full Implementation Plan
A phased, deliverable-by-deliverable plan for the Next.js front end that explains the product, captures leads, and (later) exposes a gated dashboard over the existing SEO Audit backend.
Stack: Next.js 15 (App Router) + TypeScript (strict) · Tailwind CSS · shadcn/ui · dark theme, near-black + violet accent (cbcl.ai-inspired) · deployed on Vercel.
Placeholder product name: Auditly — defined as one constant, trivially renameable.
Sequencing: Phases 0–3 ship the marketing MVP with zero dependency on backend state. Phases 4–5 add the dashboard once the backend access story is decided. Each phase is independently shippable.

PHASE 2 — How It Works Page
Goal: Set clear expectations — the process and the result a user receives.
Sections:

Intro — one-line framing: what an audit is and what you walk away with.
Pipeline detail — the five stages as end-user-facing blocks (not internals), each with icon, title, 1–2 sentence explanation:

Crawl — we fetch every page, link, image, and meta tag.
Enrich — we map the link graph, redirects, and canonical relationships.
Analyze — ~31 checks flag issues and rank them by severity.
Performance — Core Web Vitals via Google PageSpeed.
Report — a styled Excel spec your developers can action.


Checks overview — the ~31 checks grouped into categories (Metadata, Canonical/Indexing, Links, Images, Performance, i18n) as a clean card grid or definition list — communicates breadth without overwhelming.
What the report looks like — sample screenshot of the categorized ТЗ tables + a downloadable sample .xlsx (public/media/sample-report.xlsx, placeholder file until real one lands).
Expectations strip — short: turnaround, that output is an Excel file, that it's prioritized by severity.
Closing CTA → /contact.

New components: how-it-works/{stage-list,checks-grid,report-preview,expectations}.tsx; reuse Section, Card, MediaFrame, CTAButton. Checks data lives in lib/checks.ts (array of {category, items[]}), so the list edits in one place.
Acceptance: a first-time visitor understands process + output with no prior context; sample report downloads; checks render from checks.ts; fully responsive; CTA routes.

PHASE 3 — Contact Page
Goal: Capture leads / inquiries. The MVP's conversion endpoint — marketing site is fully shippable at end of this phase.
Deliverables:

Form fields: name, email (required, validated), optional site URL, message. Client + server validation with zod.
Submission via Server Action → (a) email via Resend and/or (b) insert into a leads table (Postgres/Vercel Postgres or Supabase — whichever is provisioned). Keep the storage adapter behind a small lib/leads.ts so the backend choice is swappable.
Anti-spam: hidden honeypot field + basic per-IP rate limit; reject bots silently.
States: loading, success (toast via sonner + inline confirmation), error.
Privacy note under the form.

New components: contact/contact-form.tsx (client component), app/contact/actions.ts (Server Action), lib/leads.ts (storage adapter), lib/validation.ts (zod schema shared client/server).
Acceptance: valid submit sends/stores a lead; invalid input shows field errors; honeypot rejects bots; rate limit blocks rapid repeats; success and error states both visible; no PII in logs.

Milestone: Phases 0–3 = a complete, live-able marketing site. Deploy to Vercel and start receiving traffic. Phases 4–5 are the dashboard and depend on the backend access decision.


PHASE 4 — Auth & Dashboard Shell
Goal: "Sign in" leads to a real, gated dashboard surface, with a safe path to the auth-less backend.
Deliverables:

Auth: Clerk (fastest for MVP) or Auth.js — sign in/out, session, protected route group app/(dashboard)/.
Nav "Sign in" becomes "Dashboard" when authenticated.
Backend access via server-side proxy (critical): the browser never calls the SEO backend directly. Add Next.js route handlers under app/api/proxy/* (or server actions) that run server-side, attach a shared secret/header, and forward to the backend. This lets you ship against the current auth-less backend without exposing it.
API client layer lib/api/ — typed wrappers for the backend endpoints, called only from server code.
Dashboard shell: layout, empty states, loading skeletons, dark-theme consistent.

New structure:
app/(dashboard)/ layout.tsx · page.tsx (overview)
app/api/proxy/[...path]/route.ts   # server-only forwarder w/ shared secret
lib/api/ client.ts · types.ts
middleware.ts                      # auth guard on (dashboard)
Acceptance: unauthenticated users redirected from dashboard routes; authenticated users reach an empty dashboard; all backend traffic goes browser → Next.js server → backend, never browser → backend; shared secret injected server-side only.

PHASE 5 — Audits Dashboard (Live Data)
Goal: Authenticated users see and manage their audits over the proxied backend.
Deliverables:

Audits list (GET /audits) — paginated table: URL, status, severity rollups, created date; empty state with a "Start audit" CTA.
Audit detail (GET /audits/:id) — status + finding rollups by severity (severity badges in violet/severity color scale).
Findings view (GET /audits/:id/findings) — filter by severity + rule, paginated; each finding shows rule, severity, affected URL, detail.
Report download (GET /audits/:id/report) — streamed through the proxy so the backend stays hidden.
Start audit (POST /audits) — gated behind auth and a server-side URL allowlist/validation (block private IPs/localhost, require valid public http(s) URL) to avoid SSRF, since the backend doesn't validate yet.
Polling or revalidation for in-progress audit status (the backend has no queue/websocket — poll GET /audits/:id on an interval while status is non-terminal).

New structure:
app/(dashboard)/audits/ page.tsx (list) · [id]/page.tsx (detail) · [id]/findings/page.tsx
components/dashboard/ audits-table · severity-badge · findings-table · start-audit-form · report-download-button
lib/api/ (extend with audits + findings + report methods)
Acceptance: signed-in user starts an audit (URL validated server-side), watches status update via polling, browses filtered findings, downloads the report — all proxied, no direct browser→backend traffic; SSRF-prone URLs rejected before reaching the backend.

Cross-Phase Principles
Hold these throughout so iteration stays cheap and the MVP ships safely:
Swappable content. All copy in copy.ts, nav/product name in constants.ts, checks in checks.ts, lead storage behind lib/leads.ts — iterating wording, branding, or providers never means hunting through JSX.
Placeholder-first media. MediaFrame renders correctly-sized placeholders so Phases 0–2 ship presentable before any real screenshot exists; assets drop into public/media/ with zero code change. Capture real screenshots + the sample report from the covecta.io run when convenient.
Server-side backend boundary. From Phase 4 on, the browser never touches the SEO backend directly. The Next.js server proxy (with shared secret) plus server-side URL validation is what lets you safely put a dashboard in front of an auth-less, queue-less backend — deferring backend hardening (auth, SSRF protection, job queue) without blocking the front end.
Ship the marketing site first. Phases 0–3 are fully decoupled from backend state; deploy and start receiving traffic before the dashboard exists.
